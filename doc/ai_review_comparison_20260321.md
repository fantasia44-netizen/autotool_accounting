# GPT vs Gemini 교차 비교 — 2026-03-21

> **목적**: 두 AI의 검토 결과를 비교하여 최종 액션 플랜 도출
> **결론**: 핵심 방향은 일치, 세부 전략에서 차이 존재

---

## 1. 합의 사항 (둘 다 동의)

| 항목 | 합의 내용 |
|------|-----------|
| AutoTool God Object | db_supabase.py 분리 필수, 전면 리팩토링은 금지 |
| PackFlow 구조 | 3계층 구조 적절, 과한 추상화 아님 |
| P0 보안 3건 | 즉시 수정 필수 (RLS/테넌트격리, API Key HMAC, 재고 멱등성) |
| 수식 평가기 | 블랙리스트 불충분 → 화이트리스트/AST 파서 교체 |
| 테스트 부재 | 위험 — 핵심 로직 테스트 즉시 도입 |
| MVP 삭감 | 2FA, 다중택배사, 고급통계는 베타 후 |
| PackFlow 우선 | AutoTool 안정화, PackFlow 출시에 집중 |

## 2. 의견 차이 — 상세 비교

### 2-1. 시간 배분

| | GPT | Gemini | 최종 판단 |
|--|-----|--------|-----------|
| AutoTool | 30% | 10% | **20%** — AutoTool도 숨은 보안 리스크(GPT 지적) 있으므로 10%는 위험. 크리티컬 버그+최소 테스트에 20% 배분 |
| PackFlow | 70% | 90% | **80%** |

### 2-2. 역분개 우선순위

| | GPT | Gemini |
|--|-----|--------|
| 등급 | **P0.5** (상용화 차단) | P1 (2주) |
| 근거 | "역분개 없으면 엑셀로 되돌아감" | "Append-only로 구현" |
| **최종**: **P0.5 채택** — GPT 근거가 더 강함. 3PL 월말 정산 조정은 필수 |

### 2-3. 듀얼모드 개선 시점

| | GPT | Gemini |
|--|-----|--------|
| 제안 | 베타에선 보수적 유지 | 웨이브 분할 (P2) |
| 추가 | 최소 검증 3가지 유지 | 중량계 API 연동 |
| **최종**: **GPT 채택** — 1인 개발 현실상 보수적 유지가 안전. 중량계/웨이브는 백로그 |

### 2-4. AutoTool 보안

| | GPT | Gemini |
|--|-----|--------|
| 언급 | "숨은 리스크 높음" 경고 | 직접 언급 없음 |
| 근거 | 단일 DB + 테스트 0건 = 누락 보장 불가 | — |
| **최종**: **GPT 채택** — AutoTool도 tenant 격리 점검 필요. 별도 보안 감사 항목 추가 |

### 2-5. 테스트 범위

| | GPT | Gemini |
|--|-----|--------|
| 범위 | 10개 (각 5개) | 2개 (과금+재고만) |
| 모킹 | fake repository | requests-mock |
| CI | pytest + ruff/flake8 | pytest만 (20줄) |
| **최종**: **GPT 채택(10개)** — 2개는 너무 적음. 멀티테넌트/모드판별/출고가드도 핵심 |

### 2-6. 고객포털

| | GPT | Gemini |
|--|-----|--------|
| 전략 | 조회 중심만 정리 | "세일즈 포인트" — 2주 투자 |
| **최종**: **Gemini 채택** — 고객사 대시보드가 3PL SaaS 핵심 차별점. 조회+과금 내역은 완성 |

### 2-7. db_supabase.py 분리 방법

| | GPT | Gemini |
|--|-----|--------|
| 방법 | 신규 기능만 Repository 분리 | Mixin 패턴으로 파일 분리 |
| **최종**: **Gemini 채택(Mixin)** — 기존 코드 영향 최소화하면서 파일 물리적 분리 가능 |

### 2-8. PackFlow 서비스 확장

| | GPT | Gemini |
|--|-----|--------|
| 제안 | billing 4분할 (rate_matcher, formula_eval, invoice_builder, reversal) | Pydantic 모델 도입 |
| **최종**: **둘 다 채택** — billing 분할은 코드 성장 대비, Pydantic은 타입 안정성 |

## 5. 양쪽 모두가 놓친 것 (Claude 보충)

| 항목 | 내용 |
|------|------|
| **에러 모니터링** | 양쪽 모두 Sentry/로깅 언급 약함. 베타 전 최소 Sentry 무료 플랜 연동 권장 |
| **백업 전략** | Supabase 무료 플랜의 백업 주기/복원 절차 확인 필요 |
| **rate limiting (PackFlow)** | auth.py의 인메모리 rate limit은 재시작 시 초기화 — Redis 없으면 최소 DB 기반으로 전환 |
| **CORS 설정** | API 엔드포인트 공개 시 CORS 정책 필요 |
| **환경 분리** | staging/production 환경 분리 없음 — 최소 Supabase 프로젝트 2개 권장 |
