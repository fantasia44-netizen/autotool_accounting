## 1. 아키텍처

### AutoTool: 양호하지만 구조적 위험
- db_supabase.py 5,051줄 = **확실한 God Object**
- 전면 리팩토링 금지 → **신규 기능부터만 Repository 분리**
- 분리 우선순위: orders → stock → settlement → marketplace → accounting

### PackFlow: 구조가 더 건강
- Blueprint → Service → Repository 3계층은 **적절한 수준** (과한 추상화 아님)
- 서비스 비대화 방지 필요: billing은 향후 4분할 권장
  - `rate_matcher`, `formula_eval`, `invoice_builder`, `reversal_service`

### 공통화 가능 영역
- **공유 가능**: 인증/권한, Supabase client 초기화, retry/backoff, tz 유틸, 감사로그, 파일 업로드, 엑셀 파서
- **분리 유지**: 주문/재고/정산 핵심 도메인 (공통화 욕심 내면 둘 다 느려짐)

## 2. 보안

### PackFlow P0 3건 — 베타 전 필수
1. **멀티테넌트 격리**: WHERE id AND operator_id 강제 → 공통 함수로 감싸기
2. **API 인증**: API Key + HMAC (JWT는 세션용, OAuth2는 과함)
3. **재고 이중커밋**: `reservation_id` 기준 멱등키 → DB unique 제약 + RPC "이미 committed면 no-op"

### AutoTool도 안전하지 않음
- 단일 DB 파일 + 테스트 0건 + 직접 REST 호출 → **tenant/auth 누락을 사람이 보장 불가**
- 주문수집→재고차감→정산 흐름에서 롤백/멱등성 강하게 확인 필요
- "실운영 중이라 더 위험하다"

- 운영자 사용에는 템플릿 UI, 우선순위, 중복적용 규칙, 역분개 필요

## 4. 듀얼모드

### 평가: 시장 이해가 들어간 좋은 구조
- 보수적 강등 = 초기에는 유지
- 향후: order line / SKU group 단위 hybrid 처리 → 백로그
- **1인 개발로 지금 당장 구현하면 위험**

### 속도모드 추가 검증
- 전체 검수는 빼되 **최소 검증 3가지 유지**:
  1. 송장-주문 존재성
  2. 취소 여부
  3. 합포 수량 sanity check
- "속도모드 핵심은 검수를 없애는 게 아니라 **값싼 검증만 남기는 것**"
